# Backstage Scorecard Mapping

The previous Port experiment used two scorecards: Catalog Quality and IKS Baseline. Backstage does not consume those Port scorecard YAML files directly.

## MVP Implementation

The MVP implementation uses a **local/CI Markdown report** generated by a Python checker. This approach fits the repository's docs-and-configuration boundary and allows scorecard visibility without committing a generated Backstage application.

Future options remain open:
- Backstage Tech Insights checks, if the selected Backstage distribution supports the required checks with acceptable effort.
- A maintained Backstage scorecard plugin, if it fits the MVP without introducing heavy custom development.

The MVP behavior remains advisory. Checks create visibility and follow-up signals, but they do not block teams, provision infrastructure, or make compliance decisions.

## Check Definitions

Executable check mappings are defined in `checks.yaml` with two groups:

### Catalog Quality (6 checks)

| Check | Title | Backstage field |
|---|---|---|
| `has-technical-owner` | Technical owner is assigned | `spec.owner` |
| `has-system` | System is assigned | `spec.system` |
| `has-repository` | Repository is linked | `metadata.annotations.backstage.io/source-location` |
| `has-lifecycle` | Lifecycle is classified | `spec.lifecycle` |
| `has-documentation` | Documentation link is present | `metadata.annotations.backstage.io/techdocs-ref` or documentation link |
| `has-runbook` | Runbook link is present | `metadata.annotations.iks.dev/runbook-url` or runbook link |

### IKS Baseline (5 checks)

| Check | Title | Backstage field |
|---|---|---|
| `has-business-owner` | Business owner is assigned | `metadata.annotations.iks.dev/business-owner` |
| `has-criticality` | Criticality is classified | `metadata.annotations.iks.dev/criticality` |
| `has-protection-need` | Protection need is classified | `metadata.annotations.iks.dev/protection-need` |
| `has-data-class` | Data class is classified | `metadata.annotations.iks.dev/data-class` |
| `has-iks-scope` | IKS scope is declared | `metadata.annotations.iks.dev/compliance-scope` contains `iks` |

## Local Usage

Install PyYAML dependency:

```bash
python3 -m pip install PyYAML
```

Generate a report locally:

```bash
python3 backstage/scorecards/check_catalog_scorecards.py \
  --catalog-root backstage/catalog/locations.yaml \
  --checks backstage/scorecards/checks.yaml \
  --output /tmp/catalog-scorecard-report.md
```

Validate expected demo fixture behavior:

```bash
python3 backstage/scorecards/check_catalog_scorecards.py --assert-demo-fixtures
```

## Expected Demo Results

- **`customer-portal`**: Passes all 11 checks (complete reference service)
- **`reporting-api`**: Fails only `catalog-quality/has-runbook` intentionally to demonstrate catalog quality gaps

## Interpreting Results

- **`PASS`**: The mapped Backstage metadata field is present and non-empty
- **`FAIL`**: The service needs follow-up metadata work

Failures are advisory signals for catalog improvement. They should lead to GitHub issue workflows (see issue #22), not enforcement or deployment blocking.

## CI Integration

The GitHub Actions workflow `.github/workflows/validate-idp-config.yml` generates an advisory report on every pull request and push to main. The report appears in the GitHub Actions summary for reviewer visibility.

Report generation itself only exits nonzero for execution problems (unreadable files, invalid YAML, missing expected demo services). Advisory check failures do not fail the workflow.
